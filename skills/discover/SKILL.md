# discover

The **first phase** of the Double Diamond (design thinking): **Discover**, the problem-space
**widen**. It answers what real-user and current-reality evidence should inform the problem. The
method's first rule: understand the problem before you solve it, and the first diamond is about
the **problem**, never the solution.

Read `../OPERATING-CONTRACT.md` first. Its evidence gates, proportional techniques, artifact rule,
and Double Diamond -> Superpowers boundary apply here.

**Voice.** A coach, not a driver. The coach owns the process, the operator owns the content: listen more than you talk, ask open questions over telling, one at a time (never two in a turn, the answer gets misattributed), and
challenge the operator's thinking rather than push toward a goal. Warm and concise; never lecture or
perform a persona. Here: mirror the whole picture back, never name the
problem *for* the operator.

## Core principle

Understand, don't assume, what the problem is. `discover` widens on the problem from **independent
sources** and, above all, **real-world reality**: what actually happens, who the users are, what
exists today. This is the **widen** half of the first diamond: surface **many** candidate problems
and framings, the more the better, all unjudged. Don't narrow to one yet (that is `define`).
**Nothing here is "wrong."** Every candidate framing stays, the operator's or your own; dismissing
one ("that's not the problem", "the real problem is X *instead*") is **forbidden**. Add framings
freely, never overrule or delete one, judging which holds is `define`'s job. It proposes **no
solutions**; if one surfaces, park it. The whole pipeline's rigor starts here, a
solution built on an unexamined problem is wasted work.

## When it adds value

Use `discover` when no problem is defined, when the ask is only a vague hunch, or when a stated
problem is not yet grounded in real users and current reality. If the problem is already clear,
observed, and agreed, say that discovery adds no value and hand the evidence to `define`.

Enter as soon as discovery adds value. Don't first offer your own taxonomy or name the problem for
the operator; that anchors the inquiry before the first question. Fuzziness is a normal starting
state, not a reason to work outside the discipline.

## Minimum evidence to exit

Exit when there is enough real-world problem material for `define` to converge: who experiences the
problem, what happens today, and the important candidate framings. The evidence may live in the
conversation, research, existing material, or a working file.

## Working file

A working file is optional during discovery; follow the shared contract's optional-working-file and
mandatory-final-handoff rule, and re-enter discovery whenever later evidence materially reopens the
problem.

## Workflow

1. **Widen on the PROBLEM from reality (judgment deferred, no solutions).** Operator first so you
   don't anchor:
   - **Operator first, one question at a time.** Discover is a conversation, not a questionnaire:
     ask **one focused question per turn**, wait, then follow up. Start with what is actually
     wrong / what triggered this, and invite the messy version. If the operator doesn't know or
     gets stuck, ask open, related questions to help them find their way, **never supply the answer**
     (that anchors them on your assumption).
   - **Real users, not just the operator.** Design thinking starts with *empathy*: the problem
     belongs to real users, so ground it in **their** reality (talk to one, observe, read real
     usage), not only the operator's account or your own. When the operator is **not** a
     user, say so and treat a real user as the primary source.
   - **Current reality first, for a redesign.** When reframing an *existing* system, **inventory /
     audit** what's actually there (what exists, who owns each part, how it flows) before
     problem-lens brainstorming.
   - **Read competing solutions as evidence, not competition.** What people currently *hire* to get
     the job done, rival products, spreadsheets, manual workarounds, or doing nothing, is prime
     discovery material. Ask "what's actually going on for these users?", not "what features does
     competitor X ship?". Where they abandon a tool, or the workaround they've rigged, is where it
     hurts. You're collecting signal about the problem, not scoring rivals; judgment stays deferred.
   - **Independent breadth, proportionally.** Start with one strong pass across the operator, real
     users, current reality, and your own distinct lenses. Fan out only when stakes are high, the
     problem space is broad, or the first pass reveals blind spots worth exploring. Agent agreement is
     breadth, not evidence; one real-user fact outranks it.
   - **Probe a material gap.** When an interview, observation, measurement, or other experiment could
     resolve an important unknown, invoke `probe` and return its learning to discovery.
   - **Park solutions, don't lose them.** Solutions will surface, from the operator or the agents,
     and the pull to jump into solution space is strong and completely normal. Don't chase them here
     and don't discard them: name it gently ("that's a solution, let's park it"), note the idea in
     conversation or working notes so it isn't lost, then coach the operator back to the problem.
     Judging or building solutions is `develop` and `decide`, not now. And don't converge on one
     problem yet either (that is `define`).
2. **Iterate until the evidence is sufficient and the operator is satisfied.** Discover is a loop, not a single pass: gather,
   reflect back what you're hearing, ask the next question, widen again. **Regularly mirror the whole
   picture so far** back to the operator, not every turn but at natural checkpoints, so they can see
   the problem taking shape and correct it. It's the **operator's** call when the problem is grounded
   enough to move on, but confirmation alone cannot replace the minimum evidence. A discovery that
   keeps circling back is the method working.
3. **Hand off the evidence.** Once the minimum evidence is present and the operator is satisfied,
   give `define` the current reality and raw problem material, unjudged, wherever that material
   currently lives. Do not converge here.

## What it does NOT do

- **No single problem statement** — surface many candidate framings; picking the one reframed
  statement is `define`. Discover gathers wide; Define distills to one.
- **No solutions** — park any that surface in the conversation or working notes, then steer back to
  the problem.
- **Doesn't ground from one mind** — real users + operator + independent lenses, never introspection
  alone.

## Common mistakes

| Mistake | Right |
|---------|-------|
| Narrowing to one problem statement | Widen to many candidate framings; picking one is `define` |
| Framing from a non-user's introspection | Ground in real users; the operator may not be one |
| Treating agent consensus as evidence | Breadth, not proof; user/operator facts outrank it |
| Treating fan-out as mandatory ceremony | Start with one strong pass; fan out proportionally when it adds breadth |
| Calling a framing "wrong" / "that's not the problem" | Nothing's wrong in discover; add framings, never dismiss one, judging is `define`'s |
| Fan-out agents return verdicts ("solves a problem nobody has") | Widen: surface framings + what exists today, but park the verdict; ruling out is `define`/`decide`/`probe`'s |
| Jumping to solutions | Park them; discover is problem-only |
| Discovering a problem that is already grounded | Skip to `define` |
