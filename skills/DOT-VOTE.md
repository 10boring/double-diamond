# Dot-vote (optional convergence technique)

The two **narrow** phases may use this technique: `define` for problem framings and `decide` for
solution mechanisms. It surfaces priority across a clustered field before the operator picks. Use
it only when there is a genuinely wide field or multiple stakeholders need structured
participation. When direct reasoning is sufficient, skip the vote and ask the operator to confirm
the reasoned choice.

**You facilitate it, don't hand the operator a menu of how to run it.** The dot budget and whether
to bring in blind agent-voters are *your* calls as facilitator; keep it light. The operator's input
is their dots and the final pick, not the machinery. (You drive the process; they drive the content.)

## How to run it

1. **Group first.** Cluster the raw items (framings or options) into a handful of distinct ones
   (affinity), so the vote is over real alternatives, not restatements of one.
2. **Operator votes first.** Give the operator a dot budget (default 5) to spread across the
   clusters, stacking allowed. Collect the operator's dots **before** showing any panel tally, so
   the panel never anchors them.
3. **Optional agent-voters.** Fan out independent agent-voters, each blind to the others, casting the
   same budget under the agreed criteria; then tally. This pulls the vote from independent sources,
   not one context. (Agent agreement is breadth, not evidence.)
4. **Narrow to a shortlist.** The board surfaces the top-voted few.

## The one rule

The tally **informs, it does not decide.** A tally is not a verdict: the operator holds the final
pick (and may combine items), never a default-on-silence. The vote surfaces priority; the *choice*
is reasoned and the operator's.
